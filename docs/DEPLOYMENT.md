# Развёртывание и хранение

> RuServiceBench — это **стенд, а не сервис**. «Деплой» = встроить в CI и релизный гейт ЦАМ,
> а не держать запущенным 24/7.

## Где хранить

- **Git-репозиторий.** Артефакты прогонов (`out/`) и секреты (`.env`) — в `.gitignore`.
- **Версионируется:** код, `tasks/seed_tasks.json` (изменения через PR + ревью предметника),
  `docs/`, `.github/`.
- **Секреты** — в секретах CI / vault, никогда не в коде. `.env.example` — шаблон без значений.
- **История прогонов** (если нужен тренд) — архивировать `summary.json` по дате во внешнее
  хранилище (S3/MinIO) или БД; в git артефакты не кладём.

## CI (уже настроен)

`.github/workflows/ci.yml` — на каждый push/PR гоняет unittest + dry-run (регрессия движка
без ключа и сети).

## Регресс с реальной моделью

Отдельный job (по расписанию или вручную) с ключом из секретов:
прогнать фиксированный набор → сравнить pass^k с предыдущим прогоном → завалить сборку при
просадке выше порога.

## Окружения

`dev` (fake) → `staging` (reference + ключ) → `pre-release` (перед релизом ЦАМ).
Провайдер и судья переключаются через `.env` без правок кода.

## Запуск по расписанию

GitHub Actions `schedule` (например, ночью) → артефакты портала (`index.html`) публиковать на
внутреннюю страницу/Pages для стейкхолдеров.

## Контейнер (опционально)

Ядро на stdlib, Dockerfile не обязателен. При необходимости — `python:3.12-slim` +
`pip install -r requirements.txt`, entrypoint `python run_eval.py`.
