# Пояснительная записка — RuServiceBench

Документ для всех участников: что это, на чём основано, что доказывает и где границы.
Краткая версия для руководства — [EXECUTIVE_SUMMARY.md](EXECUTIVE_SUMMARY.md).

## 1. Что это и какую задачу решает

RuServiceBench — офлайн **измерительный стенд** для оценки **надёжности** ИИ-агентов
госуслуг на русском языке (ЖКХ/ЕИРЦ, образование/кружки, документы/МФЦ). Меряет не
«получилось ли разово», а «получается ли **каждый раз**».

Проблема: средний успех («в 8 из 10 диалогов агент справился») для госуслуг обманчив.
Если услуга проходит не каждый раз — это неоплаченная квитанция, незаписанный ребёнок,
поток жалоб и повторных обращений. Нужна метрика повторяемости и стенд, ловящий
нестабильность **до** продакшена.

## 2. Чем это НЕ является

- Не продакшн-сервис: стенд запускается в CI и перед релизом, не крутится 24/7.
- Не ходит в боевые ведомственные API (почему — §4).
- Не доказывает прод-эффект: для этого отдельный трек A/B в проде.

## 3. Архитектура: что реальное, что копия, а что — продукт под тестом

Один прогон задачи — петля из четырёх частей:

| Часть | Что это | Статус |
|---|---|---|
| **Агент под тестом** | СМЕННЫЙ слот | сейчас reference (GPT-5.5+ReAct) — дублёр; ЦАМ 1.0/2.0 втыкается сюда же |
| **Мок-песочница** ЖКХ/EDU/МФЦ | контролируемая копия услуг (состояние: счета, кружки, заявки) | наша реализация |
| **Симулятор пользователя** | играет жителя, раскрывает данные по запросу (как в tau-bench) | GPT-5.5 |
| **Судья + сверка состояния** | считает успех: STATE или JUDGE | GPT-5.5 / эвристика в dry-run |

Аналогия — краш-тест: машина (ЦАМ) это продукт; трек с манекенами и датчиками (стенд) —
не настоящая дорога, но позволяет **повторяемо** мерить безопасность до выезда на улицы.

## 4. Почему мок-песочница, а не боевые API

Чтобы измерить «получилось ли», надо **задать** начальное состояние («по счёту долг 3200»)
и **проверить** финальное. На боевом ЕИРЦ так нельзя, и недопустимо, чтобы агент реально
списывал деньги во время теста. Поэтому в тестовом контуре инструменты агента
перенаправляются на нашу песочницу (или staging-копии ведомств). Сами **сценарии**
(передать показания, оплатить, записать в кружок, заказать справку) — настоящие.

## 5. Главная метрика — pass^k

- **pass¹** — обычный успех (решено хотя бы раз).
- **pass^k** — доля задач, решённых во ВСЕХ k попытках (повторяемость).
- Разрыв pass¹→pass^k = нестабильность; рядом 95% доверительный интервал (отделяет
  результат от шума).
- Несмещённая оценка `C(c,k)/C(n,k)`; число прогонов n берётся больше k (по умолчанию n=10, k=5).

## 6. Зачем reference-baseline, если это не ЦАМ

1. **Откалибровать стенд** — убедиться, что задачи решаемы, судья справедлив, метрика считается.
2. **Точка отсчёта** — «что даёт обычный сильный агент»; ЦАМ потом сравнивается с ней.
3. **Единственное, что запускается сегодня** — ЦАМ 1.0 нужен endpoint, 2.0 не запущен.

## 7. Ценность по ролям (вкладки портала `out/index.html`)

- **Руководство** — «Обзор» + «Сравнение версий»: число для приёмки 2.0, а не мнение.
- **Продукт** — «Надёжность», «Провалы»: где проседает, почему, мультидомен/проактивность.
- **Технари** — «Провалы» (диалоги), «Задачи», pass^k по задачам.
- **Эксплуатация/финансы** — «Стоимость»: токены/задержка/вызовы.
- **Предметники** — «Задачи»: каталог сценариев, правится без кода.

## 8. Что доказывает и честная граница

Доказывает корректность, кросс-доменную оркестрацию и проактивность **на сценариях** в
контролируемой песочнице. Не заменяет боевой A/B в проде — это комплементарный трек.

## 9. Статус

Готово: движок (агент+симулятор+судья), метрики надёжности, единый портал, 28 задач по
3 доменам, автотесты и CI. Reference прогоняется на GPT-5.5. Дальше: ЦАМ 1.0 (baseline,
нужен endpoint), ЦАМ 2.0 (приёмочный контур, когда запустится).
