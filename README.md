# RuServiceBench

Eval-харнесс для агентов госуслуг в стиле **τ-bench / τ²-bench**, адаптированный под **русский язык** и архитектуру GovTech (агент-координатор A2A + Доменные агенты + Skill-модели + MCP). Меряет агента на реальных русских сценариях: успех по финальному состоянию, **надёжность pass^k**, мультидомен и проактивность.

Референс методологии: `sierra-research/tau2-bench`.

## Что доказывает (привязка к R&D-тезисам)

| Тезис | Чем закрывается в харнессе |
|---|---|
| Собственный русский eval | Вся песочница: домены `jkh / edu / mfc / transport`, задачи на русском, награда по финальному состоянию |
| Надёжность, а не средний успех | Раннер гоняет каждую задачу k раз → метрика **pass^k** (доля задач, решённых во ВСЕХ k запусках) |
| Кросс-доменная оркестрация + проактивность | Типы задач `multi` (координатор задействует >1 агента) и `proactive` |

Честная граница: офлайн-харнесс доказывает оркестрацию и проактивность на сценариях. Боевой **A/B в проде** — отдельный трек прод-телеметрии, не часть этого репозитория.

## Контекст версий (важно)

- **GovTech 1.0** (классификаторы + зашитая логика) — работает сейчас → используем как **baseline** (точка «до»). Обычно оценивается в `judge`-режиме (по диалогу/достижению цели), т.к. не вызывает наши инструменты.
- **Референсный агентный baseline** (ReAct + наши инструменты на фронтир-модели) — промежуточное сравнение «что даёт агентный подход» до запуска 2.0.
- **GovTech 2.0** (A2A) — ещё не запущен. Набор задач + метрики служат **приёмочной спецификацией**: «2.0 готова, когда обходит 1.0 и закрывает gaps (мультидомен/проактивность/pass^k) на X%». Подключается адаптером, когда запустится.

## Песочница (sandbox) — обязательное условие

Чтобы мерить «по финальному состоянию», инструменты агента под тестом должны быть перенаправлены на **мок-бэкенды** этого репозитория (или staging-копии ведомств). Боевые API дёргать нельзя: не задать начальное состояние, не проверить результат, риск реальных мутаций.

## Быстрый старт (dry-run, без ключей и сети)

Ядро и dry-run работают на стандартной библиотеке Python (3.10+), установка не нужна.

```bash
python run_eval.py --dry-run        # n=10 прогонов/задача, кривая pass^k до k=5
# -> out/index.html (единый портал), out/report.md, out/runs.csv, out/summary.json
```

Dry-run использует `DemoAgent` (исполняет демо-решения задач с имитацией сбоев) и `FakeLLM` — чтобы проверить пайплайн и увидеть пример отчёта с pass^k. Это **не метрика качества**, а проверка конвейера.

Реальные прогоны (после реализации адаптеров и настройки `.env`):

```bash
cp .env.example .env   # заполнить LLM_API_KEY (и endpoints GovTech при наличии)
python run_eval.py --agent reference --runs 10   # референсный агентный baseline (готов)
python run_eval.py --agent gov1 --runs 10        # baseline: старый GovTech 1.0 (нужен endpoint)
python run_eval.py --agent gov2 --runs 10        # GovTech 2.0 (когда запустится)
# главный результат — out/index.html (единый портал)
```

## Награда: два режима

- **STATE** — сверка `task.expected` (assertions по dotted-путям) с финальным состоянием песочницы. Для референс-агента и GovTech 2.0.
- **JUDGE** — LLM-судья по цели/рубрике. Для GovTech 1.0 (без инструментов в песочнице) и для проактивности. В dry-run — простая эвристика по ключевым словам.

## pass^k

Для задачи с `n` прогонами и `c` успехами: `pass^k = C(c,k)/C(n,k)` (если `n≥k`), усредняется по задачам. `pass^1` = обычный success rate. Падение pass^k с ростом k показывает нестабильность агента — критично для госуслуг.
